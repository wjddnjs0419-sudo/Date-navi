# date-planner - 기획 문서

> 버전: 1.0.0 | 작성일: 2026-05-20 | 상태: 초안
> 레벨: 현재 프로젝트는 Starter로 감지되었으나, 기능 범위는 Dynamic 수준

## 1. 개요

### 1.1 목적

Date Planner는 커플이 함께 데이트를 제안하고, 여러 선택지를 비교하고, 의견을 나눈 뒤 최종 일정을 확정할 수 있도록 돕는 공유 플래너 앱이다.

이 앱의 목적은 채팅방에 흩어지는 데이트 계획 정보를 구조화하는 것이다. 사용자는 날짜, 시간, 장소, 데이트 종류, 후보 옵션, 상대방 피드백, 최종 결정 상태를 한 화면 흐름 안에서 관리할 수 있다.

### 1.2 대상 사용자

- 데이트 계획을 채팅이 아닌 별도 공간에서 정리하고 싶은 커플.
- 하나의 계획에 여러 후보지를 두고 비교하고 싶은 사용자.
- 확정된 데이트 일정을 가볍게 기록하고 다시 보고 싶은 커플.

### 1.3 개발 요청용 상세 프롬프트

커플을 위한 모바일 우선 Date Planner 웹 앱을 개발한다. 한 사용자는 날짜, 시간, 위치, 카테고리, 상세 설명, 여러 후보 옵션을 포함한 데이트 제안을 만들 수 있다. 연결된 상대방은 각 옵션을 확인하고, 댓글을 남기고, 수정 요청을 작성하며, 제안을 수락, 거절, 보류 상태로 변경할 수 있다. 수락된 제안은 확정된 데이트 일정으로 저장되어 공유 캘린더에 표시된다. 사용자는 고유 커플 코드를 통해 1:1로 연결된다. 프론트엔드는 React TypeScript 또는 Next.js를 사용하고, 인증, 데이터 저장, 실시간 동기화는 bkend.ai를 활용한다. UI는 따뜻한 파스텔 톤과 모바일 중심의 직관적인 화면으로 구성한다.

## 2. 목표

### 2.1 주요 목표

- [ ] 사용자가 구조화된 데이트 제안을 만들 수 있다.
- [ ] 하나의 제안 안에 Option A, Option B 같은 여러 후보 옵션을 넣을 수 있다.
- [ ] 연결된 상대방이 제안을 수락, 거절, 보류 상태로 변경할 수 있다.
- [ ] 각 후보 옵션에 댓글을 작성할 수 있다.
- [ ] 고유 커플 코드를 통해 두 계정을 1:1로 연결할 수 있다.
- [ ] 수락된 제안을 공유 캘린더 또는 일정 목록에서 볼 수 있다.
- [ ] 첫 버전은 모바일에서 쉽고 빠르게 사용할 수 있도록 만든다.

### 2.2 목표가 아닌 것

- 식당 예약, 결제, 영화표 예매 같은 외부 예약 기능.
- 두 명을 초과하는 그룹 계획 기능.
- MVP 단계의 AI 데이트 추천 기능. #google ai studio api연결 해서 구현
- 커플 외부로 공개되는 소셜 피드나 공유 기능.
- 첫 버전의 네이티브 iOS 또는 Android 앱 빌드.

## 3. MVP 범위

### 3.1 포함 범위

#### 계정 및 커플 연동

- bkend.ai 인증을 활용한 이메일/비밀번호 회원가입 및 로그인. #supanova 이용한 회원가입 및 로그인, 백엔드 구축
- 표시 이름을 포함한 사용자 프로필.
- 첫 사용자를 위한 커플 코드 생성.
- 두 번째 사용자를 위한 커플 코드 입력.
- 사용자 한 명당 하나의 활성 커플 연결만 허용.
- 한 사용자가 여러 커플에 동시에 참여하지 못하도록 하는 기본 검증.

#### 데이트 제안

- 데이트 제안 생성 항목:
  - 제목
  - 제안 날짜
  - 제안 시간
  - 주요 지역 또는 위치
  - 데이트 카테고리: 식사, 영화, 산책, 카페, 활동, 직접 입력
  - 전체 메모 또는 상세 계획
- 하나의 제안에 두 개 이상의 후보 옵션 추가.
- 각 옵션 항목:
  - 옵션 이름, 예: Option A
  - 장소 이름
  - 주소 또는 지역
  - 짧은 설명
  - 예상 비용 범위
  - 선택 입력 외부 링크
- 상대방이 응답하기 전까지 작성자가 초안 또는 대기 중 제안을 수정하거나 삭제할 수 있다.

#### 결정 상태

- 제안 상태 값:
  - `pending`: 대기
  - `accepted`: 수락
  - `declined`: 거절
- 옵션 선호도 값:
  - `liked`: 좋음
  - `neutral`: 보통
  - `not_interested`: 관심 없음
- 제안이 수락되면 선택된 옵션이 확정된 데이트 계획이 된다.
- 확정된 계획은 공유 캘린더 화면에 표시된다.

#### 피드백

- 제안 옵션별 댓글 스레드.
- 댓글 작성자, 내용, 작성 시간 저장.
- 수정 요청은 일반 댓글에 `request_change` 타입을 부여해 구분.
- 새 댓글 저장 후 댓글 목록을 갱신한다.

#### 캘린더 보기

- 모바일 친화적인 월간 보기 또는 일정 목록 보기.
- 확정된 데이트를 날짜별로 표시.
- 각 확정 일정에는 제목, 시간, 선택된 옵션, 위치를 표시.

### 3.2 MVP 제외 범위

- APNs 또는 FCM 기반 푸시 알림.
- 복잡한 실시간 접속 상태 표시.
- 지도 기반 경로 계획.
- 사진 업로드 및 앨범 기능.
- 반복 데이트 일정.
- 비용 분담 기능.

## 4. 사용자 시나리오

### 4.1 토요일 저녁 식사 데이트 제안

1. 사용자 A가 로그인하고 제안 작성 화면을 연다.
2. 사용자 A가 "토요일 저녁 강남역 맛집 데이트"라는 제목의 제안을 만든다.
3. 사용자 A가 토요일 저녁 날짜와 시간을 입력한다.
4. 사용자 A가 식당 후보 두 곳을 옵션으로 추가한다.
5. 사용자 A가 연결된 상대방에게 제안을 보낸다.

### 4.2 상대방의 확인 및 댓글 작성

1. 사용자 B가 새로운 대기 중 제안을 확인한다.
2. 사용자 B가 Option A를 열고 장소 정보를 확인한다.
3. 사용자 B가 "여기 분위기 좋아 보여!"라는 댓글을 남긴다.
4. 사용자 A는 같은 옵션 댓글 스레드에서 해당 댓글을 확인할 수 있다.

### 4.3 수락 및 일정 확정

1. 사용자 B가 Option A를 선호 옵션으로 선택한다.
2. 사용자 B가 수락 버튼을 누른다.
3. 제안 상태가 `accepted`로 변경된다.
4. Option A가 확정된 데이트 계획이 된다.
5. 해당 계획이 두 사용자 모두의 공유 캘린더에 표시된다.

## 7. 데이터 모델 초안

### 7.1 테이블 또는 컬렉션

#### `profiles`

- `id`
- `userId`
- `displayName`
- `createdAt`
- `updatedAt`

#### `couples`

- `id`
- `code`
- `ownerUserId`
- `partnerUserId`
- `status`: `waiting` 또는 `linked`
- `createdAt`
- `linkedAt`

#### `dateProposals`

- `id`
- `coupleId`
- `createdByUserId`
- `title`
- `proposedDate`
- `proposedTime`
- `locationArea`
- `category`
- `details`
- `status`: `pending`, `accepted`, `declined`
- `selectedOptionId`
- `createdAt`
- `updatedAt`

#### `dateOptions`

- `id`
- `proposalId`
- `label`
- `placeName`
- `address`
- `description`
- `estimatedCost`
- `externalUrl`
- `partnerPreference`: `liked`, `neutral`, `not_interested`
- `createdAt`
- `updatedAt`

#### `optionComments`

- `id`
- `optionId`
- `authorUserId`
- `type`: `comment` 또는 `request_change`
- `content`
- `createdAt`

### 7.2 주요 데이터 규칙

- 하나의 제안은 하나의 커플에 속한다.
- 제안은 전송되기 전에 최소 하나 이상의 옵션을 가져야 한다.
- 연결된 커플 구성원만 해당 제안을 읽거나 수정할 수 있다.
- 댓글 작성 권한은 커플 구성원 여부를 기준으로 제한한다.
- 수락된 제안은 반드시 `selectedOptionId`를 가져야 한다.

## 8. 화면 계획

| 화면 | 목적 | 주요 요소 |
|------|------|-----------|
| 로그인 / 회원가입 | 계정 접근 | 이메일, 비밀번호, 제출 버튼, 오류 상태 |
| 커플 연동 | 두 사용자 연결 | 생성된 코드, 코드 입력, 연결 상태 |
| 홈 | 전체 현황 | 대기 중 제안, 다가오는 확정 데이트, 빠른 생성 버튼 |
| 제안 작성 | 새 데이트 아이디어 등록 | 제안 폼, 옵션 편집기, 저장/전송 버튼 |
| 제안 상세 | 결정 및 의견 공유 | 상태, 옵션 카드, 댓글, 수락/거절 버튼 |
| 캘린더 | 확정 일정 확인 | 월간 보기 또는 일정 목록, 확정 일정 카드 |

## 9. UX 방향

- 홈, 작성, 캘린더, 프로필로 이동할 수 있는 모바일 하단 내비게이션을 사용한다.
- 따뜻한 파스텔 색상을 사용하되 텍스트와 버튼 대비는 충분히 확보한다.
- 대기, 수락, 거절 상태는 명확한 상태 칩으로 표시한다.
- 후보 옵션은 세로 목록으로 배치해 비교하기 쉽게 만든다.
- 댓글은 완전한 채팅이 아니라 가벼운 협업 피드백처럼 느껴지게 한다.
- MVP에서는 장식적 복잡성을 줄이고, 명확성과 빠른 계획 작성을 우선한다.

## 10. 기술 방향

### 10.1 권장 기술 스택

- Next.js App Router와 TypeScript.
- 반응형 스타일링을 위한 Tailwind CSS.
- bkend.ai 사용 범위:
  - 인증
  - 데이터 컬렉션
  - 접근 제어
  - 가능한 경우 실시간 동기화
- 서버 상태 관리를 위한 TanStack Query 또는 유사한 쿼리 훅.
- 인증과 세션 UI 상태를 위한 가벼운 클라이언트 상태 관리.

### 10.2 구현 단계

1. 프로젝트 설정 및 UI 기본 구조 구성.
2. 인증 및 보호된 라우트 구현.
3. 커플 코드 생성 및 연결 구현.
4. 제안과 옵션의 생성, 조회, 수정, 삭제 구현.
5. 댓글과 결정 상태 구현.
6. 공유 캘린더 화면 구현.
7. 실시간 갱신 또는 구독 방식 추가.
8. 시각적 완성도, 접근성, QA 점검.

## 11. 성공 기준

- [ ] 신규 사용자가 계정을 만들고 로그인할 수 있다.
- [ ] 두 사용자가 커플 코드로 하나의 커플로 연결될 수 있다.
- [ ] 연결된 사용자가 두 개 이상의 옵션을 포함한 제안을 만들 수 있다.
- [ ] 상대방이 옵션에 댓글을 추가할 수 있다.
- [ ] 상대방이 옵션을 선택하고 제안을 수락할 수 있다.
- [ ] 수락된 제안이 두 사용자 모두의 캘린더 화면에 표시된다.
- [ ] 권한이 없는 사용자는 다른 커플의 데이터를 읽을 수 없다.
- [ ] 앱은 360px 너비의 모바일 화면에서 사용할 수 있다.
- [ ] 주요 흐름에 로딩, 빈 상태, 오류 상태가 존재한다.

| MVP 범위가 지나치게 커질 수 있다. | 중간 | 높음 | 알림, 지도, AI 추천은 첫 버전 범위에서 제외한다. |
| 캘린더 UI 구현 시간이 길어질 수 있다. | 중간 | 중간 | 먼저 일정 목록 형태로 구현하고, 이후 월간 캘린더 UI를 추가한다. |
| 파스텔 디자인이 가독성을 떨어뜨릴 수 있다. | 중간 | 중간 | 밝은 배경에는 더 어두운 텍스트를 사용하고 수동으로 대비를 점검한다. |

## 13. 미확정 질문
